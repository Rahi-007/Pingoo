Key Relationships

1. User Relationships

  • User → Contact: One user can have many contacts
  • User → BlockedUser: One user can block many users
  • User → ChatMember: One user can be a member of many chats
  • User → Message: One user can send many messages
  • User → Status: One user can have many statuses
  • User → Call: One user can make/receive many calls

2. Chat Relationships

  • Chat → ChatMember: One chat can have many members
  • Chat → Message: One chat can have many messages
  • Chat → PinnedMessage: One chat can have many pinned messages
  • Chat → Group/Channel: One-to-one relationship with specialized chat types

3. Message Relationships

  • Message → Reaction: One message can have many reactions
  • Message → ReadReceipt: One message can have many read receipts
  • Message → Message (self-reference): For reply functionality

Indexes for Performance

User Table
  • email (unique) - For email-based login
  • phone (unique) - For phone-based login (WhatsApp style)
  • username (unique) - For Telegram-style username login
  • isOnline - For quick online status queries
Chat Table
  • type - For filtering by chat type
  • lastMessageAt - For sorting chats by recent activity
  • createdAt - For chronological ordering
Message Table
  • chatId - For fetching messages in a chat
  • senderId - For user message history
  • createdAt - For chronological ordering
  • isDeleted - For soft delete queries
Status Table
  • userId - For user status history
  • expiresAt - For automatic cleanup of expired statuses
  • createdAt - For chronological ordering
  • isActive - For filtering active statuses

Data Types and Constraints

String IDs

  • All primary keys use cuid() for globally unique identifiers
  • Foreign keys reference these cuid() values
  • Provides better distribution than sequential IDs

Enums for Type Safety

  • ChatType: PRIVATE, GROUP, CHANNEL
  • MessageType: TEXT, IMAGE, VIDEO, AUDIO, DOCUMENT, LOCATION, CONTACT
  • CallType: VOICE, VIDEO
  • CallStatus: INITIATED, RINGING, IN_PROGRESS, COMPLETED, MISSED, REJECTED
  • ReactionType: LIKE, LOVE, HAHA, WOW, SAD, ANGRY

JSON Fields for Flexibility

  • privacySettings in User table for customizable privacy
  • editHistory in Message table for tracking edits
  • metadata in Media table for file information
  • settings in Chat table for chat-specific configurations

Migration Strategy

Phase 1: Core Messaging

  1. User authentication and profiles
  2. Private chat functionality
  3. Basic message sending/receiving
  4. Contact management

Phase 2: Group Features

  1. Group chats with admin roles
  2. Media sharing (images, videos, documents)
  3. Read receipts and typing indicators
  4. Message reactions

Phase 3: Advanced Features

  1. Status/Stories (24-hour disappearing)
  2. Voice/video calls with WebRTC
  3. Channels and broadcast messages
  4. Bots and automation

Phase 4: Platform Integration

  1. End-to-end encryption (Signal Protocol)
  2. Marketplace features
  3. Games and mini-apps
  4. Advanced theming and customization
